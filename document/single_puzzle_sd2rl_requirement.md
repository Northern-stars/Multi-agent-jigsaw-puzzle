# 单图拼图复原（SD2RL Legacy）需求文档

> 版本: v0.1 | 更新日期: 2026-04-24
> 来源脚本: `legacy/sd2rl_torch.py`

---

## 1. 文档目的

本文档用于整理 `legacy/sd2rl_torch.py` 的功能与设计意图，并将其抽象为一份可实现、可维护的“单图复原”需求说明。该脚本代表项目早期的单图强化学习求解原型，核心目标是：

- 输入一张 `3x3` 拼图图像
- 固定中心块，仅对其余 `8` 个碎片进行重排
- 通过交换两个碎片的位置，逐步将拼图恢复为正确排列

本文档同时指出该 legacy 实现中的若干工程缺口，便于后续迁移到 `env/single_puzzle_env.py` 或新的训练框架中。

---

## 2. `legacy/sd2rl_torch.py` 功能总结

### 2.1 脚本定位

`legacy/sd2rl_torch.py` 是一个单图拼图复原实验脚本，主要包含以下能力：

- 定义单图 `3x3` 固定中心拼图环境 `Puzzle_Env`
- 定义基于局部交换的动作空间，共 `28` 个动作
- 定义 reward 计算逻辑
- 定义 DQN 风格 Agent 与经验回放
- 使用视觉编码模型 `Local_switcher_model` 对候选下一状态打分
- 提供训练主循环 `run_maze()`
- 提供测试流程 `evaluate_test_set()`

### 2.2 任务定义

该脚本处理的是单张图像的拼图复原问题：

- 原图尺寸为 `288x288`
- 被切分为 `3x3` 共 `9` 块，每块 `96x96`
- 中心块固定在位置 `(1,1)`
- 其余 `8` 个可移动碎片通过两两交换完成复原

### 2.3 状态表示

环境状态使用长度为 `8` 的排列 `index_list` 表示。

- 状态中的每个位置对应一个可移动 slot
- 值表示当前该位置放置的是哪一个可移动 piece
- 中心 piece 不参与排列编码

脚本内部大量使用“跳过中心块”的索引映射逻辑：

- 外部状态长度为 `8`
- 在计算完整 `3x3` 图像或 reward 时，会将索引 `>= 4` 的 piece / slot 向后偏移一位
- 再把中心 piece 补回位置 `4`

### 2.4 动作空间

动作空间是 `8` 个可移动位置之间的两两交换：

- 动作数为 `C(8, 2) = 28`
- 每个动作对应一对 slot 的 swap

脚本中 `env.step(a)` 的行为如下：

- 当 `a < 28` 时，执行一次交换
- 当 `a == 28` 时，不执行交换，仅用于测试流程中的“查看当前状态”

需要注意：

- 全局常量 `n_actions=28`
- 但测试逻辑显式调用了 `env.step(28)`

这意味着 legacy 代码默认存在一个“隐式 no-op / probe action”，但该动作未被纳入正式动作空间定义。这是后续工程化时必须澄清的接口问题。

### 2.5 Reward 逻辑

训练 reward 基于当前排列相对 ground-truth 的绝对质量，而不是前后差值。

当拼图完全正确时：

- 直接返回完成奖励 `ACHIEVE_REWARD = 1000`

未完成时：

- `cate`: 正确位置的 piece 数量
- `hori`: 正确水平相邻对数量
- `vert`: 正确垂直相邻对数量

综合 reward 为：

```text
reward = cate * 0.8 + (hori + vert) * 0.2 - 1.0
```

其中：

- `0.8` 对应位置正确奖励
- `0.2` 对应相邻关系奖励
- `-1.0` 为每步惩罚，鼓励更短路径完成

### 2.6 视觉模型与决策方式

Agent 并不是直接对当前状态输出 28 维 Q 值，而是采用“候选下一状态打分”方式：

1. 对当前状态枚举所有候选动作
2. 生成动作执行后的候选拼图图像
3. 将候选图像送入 `target_DQN`
4. 选择评分最高的动作

对应模型结构为：

- `Local_switcher_model`
- 主干编码器为 `fen_model`
- `fen_model` 会：
  - 将 `288x288` 图像切为 `9` 个 patch
  - 分别提取横向与纵向 patch 特征
  - 构造 `12` 个邻接 pair 特征
  - 经 MLP 得到整图表征
- `Local_switcher_model` 再输出一个标量分数

因此它更像一个“局部状态价值打分器”，而不是经典的动作值网络。

### 2.7 训练方式

脚本使用 DQN 风格训练，但形式较接近 fitted value regression：

- 经验存储为 `(s, a, r, episode_id, s')`
- 训练时重建 `s'` 对应的图像
- 使用：

```text
q_target = reward + gamma * target_DQN(s')
q_eval   = main_DQN(s')
```

- 最终优化 `MSE(q_eval, q_target)`

这套实现与标准 DQN 有明显差异：

- `q_eval` 和 `q_next` 都是对“下一状态图像”打分
- 没有显式按动作索引取 Q 值
- 更接近“学习状态价值函数 V(s)”而非“动作价值函数 Q(s,a)”

### 2.8 测试方式

测试阶段加载训练后的 `target_DQN`，循环执行：

1. 基于当前 observation 选择一个候选交换动作
2. 若执行后模型评分不下降，则接受新状态
3. 若评分下降，则停止搜索并输出当前结果
4. 超过 `TEST_STEP` 后终止

所以测试过程实际上是一个“单步爬山式局部搜索”：

- 只接受不变差的动作
- 一旦候选动作导致评分下降，则结束当前样本

### 2.9 对外部依赖的要求

该脚本依赖以下外部资源：

- 数据集：
  - `dataset/train_img_48gap_33-001.npy`
  - `dataset/train_label_48gap_33.npy`
  - `dataset/test_img_48gap_33.npy`
  - `dataset/test_label_48gap_33.npy`
- 预训练视觉模型：
  - `model/hori_{model_name}.pth`
  - `model/vert_{model_name}.pth`
- 工具函数：
  - `utils.utils.model_fen_load`
  - `utils.utils.plot_reward_curve`

此外，脚本还依赖 `hori_list / vert_list / cate_list` 与测试集对应数组，但在当前文件中它们只被初始化为全零占位数组，没有真实加载逻辑。这说明当前版本更像“历史原型”，不能直接视为完整可复现实验实现。

## 3. 单图复原需求定义

### 3.1 目标

系统需要支持单张 `3x3` 拼图的自动复原。输入为单张被打乱的拼图图像，输出为一系列交换动作，最终使 `8` 个可移动碎片恢复到正确位置。

### 3.2 任务边界

本需求只覆盖：

- 单张图像
- `3x3` 拼图
- 中心块固定
- 两两交换式复原

本需求不覆盖：

- 多图混拼
- 跨 board 交换
- 中心块可移动
- 旋转、翻转等几何变换

### 3.3 输入

每个 episode 至少包含以下输入：

- 一张 `288x288x3` 图像
- 该图像对应的 ground-truth 排列标签
- 可选的辅助评分矩阵：
  - 横向相邻评分矩阵
  - 纵向相邻评分矩阵
  - piece-slot 匹配评分矩阵

### 3.4 输出

系统需要输出：

- 当前状态下的动作选择
- episode 结束后的最终排列
- 评估指标：
  - 是否完全复原成功
  - 正确位置数 `cate`
  - 正确水平邻接对数 `hori`
  - 正确垂直邻接对数 `vert`
  - 平均步数

## 4. 环境需求

### 4.1 拼图表示

环境必须采用与现有数据一致的中心固定表示：

- 总 piece 数：`9`
- 可移动 piece 数：`8`
- 中心索引固定为 `4`
- 其余 slot 使用长度为 `8` 的排列表示

### 4.2 状态接口

环境至少应支持：

- `reset(...) -> state`
- `step(action) -> next_state, reward, done, info`

其中 `info` 至少包含：

- `cate`
- `hori`
- `vert`
- `is_success`

### 4.3 动作定义

应明确支持以下两种设计中的一种，并在实现中保持一致：

方案 A：

- 动作空间严格为 `28` 个两两交换动作

方案 B：

- 动作空间为 `29` 个动作
- 前 `28` 个为交换
- 第 `29` 个为 no-op / terminate / probe

推荐方案：

- 若环境用于强化学习训练，优先采用方案 A，避免接口歧义
- 若评估阶段确实需要 no-op，应单独定义终止动作并在文档中显式声明

### 4.4 终止条件

训练阶段至少支持以下终止条件：

- 当前排列完全正确
- 达到最大步数上限

测试阶段可额外支持：

- 价值不再提升时提前终止
- 显式 terminate action 终止

---

## 5. Reward 需求

### 5.1 基本原则

reward 应反映当前拼图状态的绝对质量，并兼顾：

- 位置正确性
- 相邻关系正确性
- 尽快完成

### 5.2 推荐 reward 结构

参考 `legacy/sd2rl_torch.py`，推荐采用：

```text
if solved:
    reward = DONE_REWARD
else:
    reward = CATE_WEIGHT * cate + PAIRWISE_WEIGHT * (hori + vert) + STEP_PENALTY
```

推荐初始参数：

- `DONE_REWARD = 1000`
- `CATE_WEIGHT = 0.8`
- `PAIRWISE_WEIGHT = 0.2`
- `STEP_PENALTY = -1.0`

### 5.3 指标定义

- `cate`：当前位置正确的 piece 数量，范围 `0~8`
- `hori`：水平相邻关系正确的 pair 数量，范围 `0~6`
- `vert`：垂直相邻关系正确的 pair 数量，范围 `0~6`

### 5.4 设计建议

后续正式实现时，reward 计算不应依赖 debug `print` 或隐式索引插补逻辑，而应通过统一的 slot/grid 映射函数完成。

## 6. 模型需求

### 6.1 输入形式

模型输入应为当前候选拼图图像或拼图状态编码。若沿用 legacy 方案，输入为：

- `B x 3 x 288 x 288` 图像张量

### 6.2 编码思路

推荐保留 legacy 脚本中的 pair-aware 编码思想：

- 将图像切分为 9 个 patch
- 提取每个 patch 的视觉特征
- 显式构造横向与纵向邻接 pair 特征
- 聚合为整图状态表征

该设计比直接做整图分类更贴近拼图任务本身。

### 6.3 输出形式

应明确选择以下两类中的一种：

方案 A：状态打分模型

- 输入候选下一状态
- 输出标量分数
- 动作通过枚举候选状态后排序得到

方案 B：动作值模型

- 输入当前状态
- 直接输出 `28` 或 `29` 维动作值

建议：

- 若希望最大限度继承 legacy 逻辑，先使用方案 A
- 若希望训练效率更高、实现更标准，逐步迁移到方案 B

## 7. 训练需求

### 7.1 数据

训练阶段需要：

- 训练图像集
- 训练标签集
- 与训练样本逐一对应的辅助评分矩阵（若 reward 或初始化依赖它们）

### 7.2 经验回放

系统应支持经验回放缓存，至少存储：

- 当前状态
- 动作
- reward
- 下一状态
- 样本索引或 episode 上下文

### 7.3 探索策略

支持 epsilon-greedy：

- 初始探索率 `epsilon_max`
- 最小探索率 `epsilon_min`
- 按 episode 衰减

### 7.4 模型更新

若使用双网络结构，应支持：

- `main network`
- `target network`
- 软更新或周期性硬更新

### 7.5 日志与产物

训练过程至少保存：

- 模型权重
- reward 曲线
- 成功率曲线
- 可选的动作分布统计

---

## 8. 评估需求

### 8.1 评估指标

评估阶段至少输出：

- Success Rate
- Average Cate
- Average Hori
- Average Vert
- Average Steps

### 8.2 评估策略

应明确采用以下策略之一：

- 固定最大步数，直到成功或超时
- hill-climbing 式，只接受评分不下降的动作
- 显式 terminate action

legacy 脚本当前使用的是第二种，但该行为应在正式实现中写入配置，而不是硬编码在测试函数里。

## 9. 工程化改造建议

基于 `legacy/sd2rl_torch.py`，后续正式单图环境建议做以下整理：

### 9.1 环境与训练解耦

将以下内容拆分：

- 环境：状态、动作、reward、done
- Agent：动作选择、经验回放、训练
- Model：视觉编码与价值头
- Evaluation：独立测试逻辑

### 9.2 去除隐式索引规则

将“插入中心块”“索引 >= 4 时加一”等逻辑收敛为统一工具函数，避免 reward、图像重建、指标计算各自维护一份映射规则。

### 9.3 补齐辅助评分矩阵加载

当前脚本中的：

- `hori_list`
- `vert_list`
- `cate_list`

缺少真实数据加载逻辑。正式版本必须：

- 明确这些矩阵是否为必需输入
- 明确生成方式与缓存路径
- 明确训练集、验证集、测试集的对应关系

### 9.4 统一动作空间定义

需要消除“`n_actions=28` 但 `step(28)` 合法”的歧义。推荐单独引入：

- `terminate_action`

或在评估器内部实现 no-op，而不是由环境隐式接受越界动作。

### 9.5 统一 reward 与指标接口

建议将以下逻辑统一为标准方法：

- `get_reward(state)`
- `get_metrics(state)`
- `is_solved(state)`

便于和 `env/single_puzzle_env.py`、`env/env.py`、`env/dual_board_env.py` 保持风格一致。

## 10. 结论

`legacy/sd2rl_torch.py` 的核心价值在于，它已经清晰给出了单图拼图复原的基础范式：

- 中心固定的 `3x3` 单图拼图表示
- 以交换为原子动作
- 以“位置正确 + 邻接正确 + 步长惩罚”为核心的 reward
- 基于视觉特征的候选状态打分

但它仍属于实验性质较强的原型代码，存在以下典型问题：

- 环境、模型、训练耦合较重
- 动作空间定义不完全一致
- 外部依赖加载不完整
- 评估策略中包含隐式规则

因此，后续如果要构建正式的单图复原模块，应以本需求文档为基准，保留其任务定义与核心 reward 思路，同时对接口、配置、数据依赖和评估流程进行工程化重构。
